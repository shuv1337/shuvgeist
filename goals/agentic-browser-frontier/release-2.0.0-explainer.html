TL;DR

v2.0.0 turns Shuvgeist from a bridge/sidepanel assistant into the foundation for an agentic browser runtime: semantic snapshots, stable refs, multi-session routing, MCP control surfaces, planner validation, memory persistence, direct-CDP headless execution, and a gated vision baseline. It also fixes narrow settings navigation and noisy reconnect behavior.

01

Before and After

Before v2.0.0

The extension and Electron bridge could run deterministic CLI commands, but the agent had limited page observation, less durable refs, a thinner bridge task model, and no no-extension direct-CDP agent runtime.

After v2.0.0

The agent gets compact semantic page state, stable element references, server-side snapshot storage, MCP task surfaces, validation hooks, memory writes, multi-window routing, and direct-CDP headless control with trusted input.

03

Change Lanes

Semantic observation and refs

Navigation now includes compact page snapshots, canonical snapshot capture moved into shared helpers, refs carry stable element ids, and snapshot filtering keeps token usage bounded.

SNAPSHOT_PAGE_SCRIPTstableElementIdquery filter

Runtime and provider foundations

Agent construction moved behind a UI-free runtime factory. Provider metadata, credential parsing, default model resolution, and the bundled free-tier path are centralized and tested.

ProviderCatalogProviderCredentialfree tier

Bridge and MCP control surfaces

The bridge gained a task lifecycle, request handler, server-side raw snapshot storage, per-handle write locks, session registry routing, sensitive cookie access plumbing, and MCP Streamable HTTP tools.

bridge serverMCPtask registry

Agent quality loop

A benchmark harness measures fixed scenarios, planner-validator hooks can detect drift and block bad tool calls, the validator is enabled by default, and memory storage records useful skill trajectory notes.

planner validatoreval harnessmemory store

Headless direct-CDP runtime

The new headless adapter connects to no-extension Chromium targets, exposes snapshot/locate/ref-click tools through the agent loop, dispatches trusted input without Runtime on the action path, and captures screenshot plus candidate JSON for vision-capable fallback.

direct CDPtrusted inputvision baseline

UX, process, and release hygiene

Settings navigation was changed for narrow sidebars, reconnect logging was quieted, spike reports and gate decisions document frontier risk, and the package/manifest versions moved to 2.0.0.

ATTENTION

New no-extension Chromium adapter. It discovers CDP page targets, builds the agent runtime with page snapshot, locate-by-role, and ref-click tools, supports reconnect by target id, captures screenshots, and builds the T13 vision candidate baseline.

ATTENTION

New direct-CDP action path for clicks. It sends only Input.dispatchMouseEvent and is unit-tested to avoid Runtime.enable on the trusted input path.

Where to Review

1
Direct-CDP action path

Check that trusted input remains limited to CDP input events and does not enable Runtime on click actions.

2
Bridge server and MCP task surfaces

This is the broadest integration area: session routing, task state, sensitive cookie access, snapshot reads, and command dispatch all meet here.

3
Planner validator default behavior

The validator is now on by default. Review the setting escape hatch and drift-blocking behavior for false positives.

4
T8 gate state

The code is present, but the process gate remains withheld because original human approval proof was not recovered.
